# 2. Map 101 – Core Geospatial Concepts

---

## 2.1 Positioning System

Earth is a rotating sphere.

- **Latitude** → North/South
- **Longitude** → East/West

Coordinates uniquely identify a location:

```
(37.423021, -122.083739)
```

---

## 2.2 3D → 2D: Map Projection

Transforming spherical Earth to flat 2D surface is called **Map Projection**.

Google Maps uses:

## Web Mercator

![Image](https://upload.wikimedia.org/wikipedia/commons/e/ec/Web_maps_Mercator_projection_SW.jpg)

![Image](https://upload.wikimedia.org/wikipedia/commons/7/73/Mercator_projection_Square.JPG)

![Image](https://upload.wikimedia.org/wikipedia/commons/thumb/d/d6/XYZ_Tiles.png/1024px-XYZ_Tiles.png)

![Image](https://support.safe.com/hc/article_attachments/25409128976909)

### Why Web Mercator?

- Preserves angles (important for navigation)
- Tiles align into perfect squares
- Easy tiling system
- Industry standard for web maps

### Trade-off

- Distorts area near poles

---

### Other Projections (Reference Summary)

| Projection                    | Key Property        | Tradeoff         |
| ----------------------------- | ------------------- | ---------------- |
| Mercator projection           | Preserves angles    | Area distortion  |
| Gall–Peters projection        | Preserves area      | Shape distortion |
| Winkel tripel projection      | Balanced compromise | Not conformal    |
| Peirce quincuncial projection | Conformal           | Complex layout   |

For navigation, **angle preservation matters most**, hence Mercator.

---

## 2.3 Geocoding

Convert address → (lat, lng)

Example:

```
"1600 Amphitheatre Parkway" → (37.423021, -122.083739)
```

### How it works

- Parse address
- Match against GIS database
- Interpolate building location
- Return coordinate

Reverse geocoding does opposite.

---

## 2.4 Geohashing

Spatial indexing technique that encodes lat/lng into short string.

Recursive space partitioning:

```
World
 ├── 00
 ├── 01
 ├── 10
 └── 11
```

Each block subdivided recursively.

Benefits:

- Nearby locations share prefix
- Easy sharding
- Perfect for tiling
- Efficient range queries

# 3. Map Rendering

---

## 3.1 Tile-Based Rendering

Instead of rendering whole world:

- Break into 256×256 pixel tiles
- Client fetches only visible tiles
- Tiles stitched into mosaic

### Zoom Levels

Google Maps uses 21 zoom levels.

![Image](https://upload.wikimedia.org/wikipedia/commons/thumb/b/bd/OSM%2C_entire_world_map_at_zoom_level_0_in_single_tile.svg/960px-OSM%2C_entire_world_map_at_zoom_level_0_in_single_tile.svg.png)

![Image](https://learn.microsoft.com/en-us/azure/azure-maps/media/zoom-levels-and-tile-grid/map-2x2-tile-layout.png)

![Image](https://learn.microsoft.com/en-us/azure/azure-maps/media/zoom-levels-and-tile-grid/quadkey-tile-pyramid.png)

![Image](https://learn.microsoft.com/en-us/azure/azure-maps/media/zoom-levels-and-tile-grid/map-tiles-x-y-coordinates-7x7.png)

- Zoom 0 → 1 tile (whole Earth)
- Zoom 1 → 2×2 tiles
- Zoom 2 → 4×4 tiles
- Each zoom → 4× more tiles

## 3.2 Raster vs Vector Tiles

Modern systems use vectors.

![Image](https://gero.dev/static/images/osm-renderer-pt1/osm-renderer-pt1-hero.webp)

![Image](https://www.ordnancesurvey.co.uk/img-assets/media-components/07-Raster-vs-Vector-smallscale.xaa788396.jpg?f=webp&w=3840)

![Image](https://media.maptiler.com/img/2019_02_07_what_are_vector_tiles_and_why_you_should_care_1_e28a134a8f.png)

### Why vectors?

- Smaller payload
- Smooth zooming
- No pixelation
- GPU accelerated via WebGL

This aligns with insights from **Google’s smoother map article**:
Move rendering to client GPU → better performance & less bandwidth.

# 4. Back-of-Envelope Estimation

---

## Storage

### Tiles at Zoom 21

- ~4.4 trillion tiles
- 100KB each
- ≈ 440 PB

After compression (90% water/desert):

≈ 50–70 PB

Safe estimate:

**~100 PB**

---

## Traffic Estimation

Assume:

- 1B DAU
- 35 mins/week usage
- 5B minutes/day

If 1 GPS update/sec:

```
5B × 60 = 300B requests/day
QPS ≈ 3M
```

Batch every 15s:

```
3M / 15 = 200K QPS
Peak (×5) ≈ 1M QPS
```

# 7. Routing & Navigation

---

## 7.1 Road Data Model

Road network = Graph

- Nodes → intersections
- Edges → roads
- Stored as adjacency list

Reference: Adjacency list

---

## 7.2 Routing Tiles

Inspired by hierarchical routing (Valhalla docs).

3 layers:

1. Local roads
2. Arterial roads
3. Highways

Stored in object storage.

Loaded on demand.

---

## 7.3 Shortest Path

Uses:

## A\* search algorithm

Why A\*?

- Faster than Dijkstra
- Uses heuristic (distance to goal)
- Perfect for spatial graphs

---

## 7.4 ETA Service

From:

- Real-time traffic
- Historical patterns
- ML models

Google research shows usage of Graph Neural Networks.

Reference summary from DeepMind traffic paper:

- Traffic modeled as graph
- Nodes = road segments
- GNN captures spatial dependencies
- Predicts future speeds 10–20 min ahead

Key insight:

Traffic is **spatio-temporal problem**, not independent edges.

---

## 7.5 Ranker

Filters:

- Avoid tolls
- Avoid highways
- Fastest
- Shortest
- User preference

Returns top-k routes.

---

# 8. Adaptive ETA & Rerouting

Naive approach:

Store route as list of tiles → O(n × m)

Optimized:

Store hierarchical parents:

```
r1 → super(r1) → super(super(r1))
```

When incident happens:

- Check high-level tile containment
- Filter most users quickly

Time reduces dramatically.

---

## Real-time Push

Options:

- Push notifications (limited)
- Long polling (inefficient)
- SSE
- WebSocket

Choose:

WebSocket (bi-directional, low latency)

# 10. CDN & Bandwidth

Assume:

- 30 km/h
- 200m × 200m tiles
- 25 tiles per km²
- 100KB per tile

→ 2.5MB/km²
→ 75MB/hour
→ 1.25MB/min

Global daily CDN traffic:

≈ 6.25B MB/day

With 200 POPs:

≈ 300MB/s per POP

---

Maps is API-driven microservices architecture.

Key insight:

- Move rendering to GPU
- Use vector tiles
- Reduce bandwidth

- Preserves local angles
- Great for navigation
- Distorts area at poles

Chosen because navigation > visual area accuracy.

- Parsing → normalization → interpolation
- Rooftop vs approximate matches
- Uses multiple data sources

Accuracy tiers matter.

---

### 5. Geohashing Article

- Base32 encoding
- Prefix similarity
- Efficient range search
- Great for distributed sharding

Core spatial indexing technique.

- Tile-based routing graph
- Memory efficient
- Load on demand
- Hierarchical levels
- Used by large-scale navigation engines

- Graph neural networks
- Predict travel time
- Spatio-temporal modeling
- Real-time updates
- Improves ETA accuracy significantly

# 14. Final System Flow

1. User sends location → Location Service → Cassandra + Kafka
2. Kafka updates traffic + ML + routing tiles
3. User requests navigation → Geocoding → Route Planner
4. A\* finds paths
5. ETA ML ranks them
6. Ranker applies filters
7. Map tiles fetched via CDN
8. Adaptive ETA via WebSocket
