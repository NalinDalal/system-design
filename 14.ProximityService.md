# Proximity Service

It is used to discover nearby places such as restaurant, hotel, threaters, museums etc.
Main component of a something similar to google maps.

---

# Design Scope

- **User-defined search radius** (default configurable).
- **Maximal radius**: 20 km.
- **UI control**: radius adjustable by user.
- **Business endpoints**: add, delete, or update business data.
- **User assumption**: moving speed is slow (no need for constant GPS streaming).

## Functional Requirements

- Return all businesses within a given `(latitude, longitude, radius)`.
- CRUD operations for business owners.
- Customers can view **detailed business info**.

## Non-Functional Requirements

- **Low latency**: fast lookup of nearby businesses.
- **High availability & scalability**.
- **Data privacy & security**.

---

# Back of Envelope Estimation

- **Seconds in a day**:

  ```
  24 * 60 * 60 ≈ 10^5
  ```

- **100M DAU**, average **5 queries/day**.

  ```
  Total queries = 100M * 5 = 500M/day
  ```

- **Average QPS**:

  ```
  500M / 10^5 ≈ 5000
  ```

- **Peak QPS**: ~10K (2x buffer).

### Storage

- Businesses: **100M globally**.
- Each record ≈ 1 KB (id, name, category, lat, lng, metadata).
- Total storage: **~100 GB**.

**GET** `/v1/search/nearby`

Params:

- `lat` → user latitude
- `lng` → user longitude

2. [Uber H3 Geospatial Index](https://h3geo.org/)
H3 is a discrete global grid system for indexing geographies into a hexagonal grid, developed at Uber.

Coordinates can be indexed to cell IDs that each represent a unique cell.

Indexed data can be quickly joined across disparate datasets and aggregated at different levels of precision.

3. [Redis Geospatial Index](https://redis.io/docs/data-types/geospatial/)

Redis Query Engine supports geospatial data — store geographical locations and geometric shapes in JSON object fields, index them, and query by location or shape relationships.

Redis uses coordinate points for locations. Individual points or polygon shapes can be stored. Coordinates can be geographical (longitude/latitude) or Cartesian (flat plane).

**Schema Types:**
- **GEO**: Simple longitude-latitude pairs (e.g. `"-104.991531, 39.742043"`). Supports basic point-and-radius queries.
- **GEOSHAPE**: Uses WKT (Well-Known Text) subset for `POINT` and `POLYGON` primitives with geographical or Cartesian coordinates. Supports advanced queries.

```
// GEO — store as "longitude, latitude" string or JSON array of such strings
{ "location": "-104.991531, 39.742043" }
{ "location": ["-104.991531, 39.742043", "-105.0618814, 40.5150098"] }

// GEO query — find within radius (longitude, latitude, radius, unit)
FT.SEARCH productidx '@location:[-104.800644 38.846127 100 mi]'

// GEOSHAPE — POINT primitive
{ "geom": "POINT (2 2)" }

// GEOSHAPE — POLYGON primitive (closed shape, double parentheses; supports holes)
{ "geom": "POLYGON ((1 1, 1 3, 3 3, 3 1, 1 1))" }

// GEOSHAPE queries — four operations:
// WITHIN:     find points/shapes entirely within an enclosing shape
// CONTAINS:   find shapes that completely contain a specified point/shape
// INTERSECTS: find shapes whose boundary overlaps another shape
// DISJOINT:   find shapes whose boundary does not overlap another shape

// Example: find shapes within a polygon (excluding "Green Square" by name)
FT.SEARCH geomidx "(-@name:(Green Square) @geom:[WITHIN $qshape])" PARAMS 2 qshape "POLYGON ((1 1, 1 3, 3 3, 3 1, 1 1))" RETURN 1 name DIALECT 2
```

4. [Quadtree Spatial Indexing ](https://en.wikipedia.org/wiki/Quadtree)

A quadtree is a tree data structure where each internal node has exactly four children, used to partition a 2D space by recursively subdividing it into four quadrants. Named by Finkel and Bentley in 1974.

**Common features of all quadtrees:**
- Decompose space into adaptable cells
- Each cell/bucket has a maximum capacity; when reached, the bucket splits
- The tree directory follows the spatial decomposition

**Types of Quadtrees:**
- **Region quadtree**: Partitions space into four equal quadrants recursively. Each leaf stores data for its subregion. Used for image representation/compression, variable resolution data fields. Height depends on spatial distribution.
- **Point quadtree**: Adapts binary tree for 2D point data. Center of subdivision is always on a point. Operates in O(log n) time. Surpassed by k-d trees for generalized binary search.
- **Point-region (PR) quadtree**: Like region quadtree but leaf cells store a list of points rather than uniform values. Height depends on spatial distribution.
- **Edge quadtree**: Stores lines rather than points. Subdivides until one line segment per cell.
- **Polygonal map (PM) quadtree**: Stores collections of polygons. Variants: PM1, PM2, PM3 (differ in what black nodes store).
- **Compressed quadtree**: Prunes empty nodes and compresses long single-child paths. Achieves O(log n) search/insert/delete using Z-order curves.

**Common uses:** image representation, image processing, mesh generation, spatial indexing, point location queries, range queries, collision detection, view frustum culling, sparse data storage.

```pseudo
// Core structures
struct XY { float x; float y; }
struct AABB { XY center; float halfDimension; }  // Axis-aligned bounding box

class QuadTree {
    constant int QT_NODE_CAPACITY = 4;
    AABB boundary;
    Array of XY points;
    QuadTree* northWest, northEast, southWest, southEast;

    // Insert a point — subdivides when capacity is reached
    function insert(XY p) {
        if (!boundary.containsPoint(p)) return false;
        if (points.size < QT_NODE_CAPACITY && northWest == null) {
            points.append(p); return true;
        }
        if (northWest == null) subdivide();
        if (northWest->insert(p)) return true;
        if (northEast->insert(p)) return true;
        if (southWest->insert(p)) return true;
        if (southEast->insert(p)) return true;
        return false;
    }

    // Query range — find all points within an AABB
    function queryRange(AABB range) {
        Array of XY pointsInRange;
        if (!boundary.intersectsAABB(range)) return pointsInRange;
        for (int p = 0; p < points.size; p++)
            if (range.containsPoint(points[p]))
                pointsInRange.append(points[p]);
        if (northWest == null) return pointsInRange;
        pointsInRange.appendArray(northWest->queryRange(range));
        pointsInRange.appendArray(northEast->queryRange(range));
        pointsInRange.appendArray(southWest->queryRange(range));
        pointsInRange.appendArray(southEast->queryRange(range));
        return pointsInRange;
    }
}
```

5. [R-Tree Spatial Indexing ](https://en.wikipedia.org/wiki/R-tree)

R-trees are tree data structures for indexing multi-dimensional information (geographical coordinates, rectangles, polygons). Proposed by Antonin Guttman in 1984. Used for queries like "find all museums within 2 km" or "find nearest gas station".

**Key idea:** Group nearby objects and represent them with their minimum bounding rectangle (MBR) at the next higher level. A query that doesn't intersect the bounding rectangle cannot intersect contained objects — enabling efficient pruning.

**Properties:**
- Balanced search tree (all leaf nodes at same depth), like B-tree
- Organized in pages with max entries M and minimum fill (30-40% optimal)
- Suitable for large datasets and databases (nodes paged to memory as needed)
- Does not guarantee good worst-case performance, but works well with real-world data

**Variants:** Priority R-tree (worst-case optimal), R\*-tree, R+ tree, Hilbert R-tree, X-tree

**Data layout:**
- Non-leaf entries: child node pointer + bounding box of all entries in that child
- Leaf entries: spatial object data (point or MBR) + external identifier

**Algorithms:**

```
// Search (range query) — Filter & Refine Principle
function search(node, queryBox):
    if node is leaf:
        return entries where entry.MBR intersects queryBox
    results = []
    for each entry in node:
        if entry.MBR intersects queryBox:
            results.append(search(entry.child, queryBox))
    return results

// Nearest neighbor — priority queue approach
function nearestNeighbor(root, queryPoint):
    queue.insert(root)
    while queue not empty:
        entry = queue.popNearest(queryPoint)
        if entry is leaf object: return entry  // found result
        if entry is node: expand and reinsert children into queue

// Insertion — choose subtree needing least enlargement
function insert(node, object):
    if node is leaf:
        if node not full: add object to node
        else: split node, add object, propagate overflow upward
    else:
        choose child whose MBR needs least enlargement
        insert(child, object)
        update MBR

// Splitting heuristics:
// - QuadraticSplit: find worst pair, assign remaining by area increase preference
// - LinearSplit: faster but worse structure
// - R*-tree split: minimizes overlap, prefers quadratic pages, uses reinsertion
// - X-tree: may create super-nodes instead of splitting (for high-dimensional data)

// Deletion — remove entry, update MBRs, dissolve underfull pages (reinsert children)

// Bulk-loading strategies:
// - Nearest-X: sort by first coordinate, split into pages
// - Packed Hilbert R-tree: sort by Hilbert value of center
// - Sort-Tile-Recursive (STR): estimate leaves needed, split dimensions recursively
// - Overlap Minimizing Top-down (OMT): improves STR by minimizing overlap
```

7. [MongoDB Geospatial Queries](https://www.mongodb.com/docs/manual/geospatial-queries/)

MongoDB supports geospatial data as **GeoJSON objects** (for spherical calculations using WGS84) or **legacy coordinate pairs** (for flat/Euclidean calculations). Longitude is always listed first.

**Geospatial Indexes:**
- **2dsphere**: Queries that interpret geometry on a sphere (GeoJSON objects)
- **2d**: Queries that interpret geometry on a flat surface (legacy coordinate pairs)

**Data Formats:**

```js
// GeoJSON Point
{ location: { type: "Point", coordinates: [-73.856077, 40.848447] } }  // [longitude, latitude]

// Legacy coordinate pairs
{ location: [-73.856077, 40.848447] }  // array (preferred)
{ location: { lng: -73.856077, lat: 40.848447 } }  // embedded document
```

**Query Operators:**

```js
// $geoIntersects — geometries that intersect with a GeoJSON geometry (2dsphere)
db.collection.find({ location: { $geoIntersects: { $geometry: { type: "Polygon", coordinates: [[...]] } } } })

// $geoWithin — geometries within a bounding GeoJSON geometry (2dsphere, 2d)
db.collection.find({ location: { $geoWithin: { $geometry: { type: "Polygon", coordinates: [[[-80,10],[-80,9],[-79,9],[-79,10],[-80,10]]] } } } })

// $near — objects in proximity to a point, sorted nearest to farthest (2dsphere, 2d)
db.places.find({ location: { $near: { $geometry: { type: "Point", coordinates: [-73.9667, 40.78] }, $minDistance: 1000, $maxDistance: 5000 } } })

// $nearSphere — objects in proximity on a sphere (2dsphere, 2d)
```

12. [GeoHashing Techniques](https://en.wikipedia.org/wiki/Geohash)

Geohash is a public domain geocode system (2008, Gustavo Niemeyer; based on Morton's 1966 Z-order curve) that encodes a geographic location into a short string of letters and digits. It is a hierarchical spatial data structure that subdivides space into buckets of grid shape.

**Key Properties:**
- Arbitrary precision — remove characters from end to reduce precision gradually
- Longer shared prefix = spatially closer (but reverse not guaranteed)
- Uses base-32 alphabet: `0-9, b-h, j-k, m-n, p-z` (excludes a, i, l, o)
- Hierarchical: each character adds precision, subdividing the cell into 32 sub-cells

**Algorithm:**
1. Interleave latitude and longitude bits (longitude = even positions, latitude = odd positions)
2. Each bit halves the search interval (longitude: -180 to 180, latitude: -90 to 90)
3. Encode the resulting binary string in base-32

**Precision Table:**

| Geohash Length | Lat Bits | Lng Bits | km Error |
|---|---|---|---|
| 1 | 2 | 3 | ±2,500 km |
| 2 | 5 | 5 | ±630 km |
| 3 | 7 | 8 | ±78 km |
| 4 | 10 | 10 | ±20 km |
| 5 | 12 | 13 | ±2.4 km |
| 6 | 15 | 15 | ±0.61 km |
| 7 | 17 | 18 | ±0.076 km (76 m) |
| 8 | 20 | 20 | ±0.019 km (19 m) |

**Limitations:**
- **Edge cases**: Points on opposite sides of 180° meridian, equator, or Greenwich meridian have no common prefix despite being close
- **Non-linearity**: Lat/lon distance ≠ actual distance (error grows from 0.67% at equator to infinity at poles)
- **Proximity search**: Bounding box approach retrieves too many points; solutions include prefix filter lists (used in Solr)

**Main uses:** Unique identifier, point data in databases (contiguous index slices for rectangular areas), quick proximity search, geotagging. Used in Elasticsearch, MongoDB, HBase, Redis, Accumulo.

**Similar systems:** Open Location Code (plus codes), S2 Geometry (Hilbert curve), what3words, MapCode, Military Grid Reference System, Maidenhead Locator System.

13. [KD-Trees for Spatial Search ](https://en.wikipedia.org/wiki/K-d_tree)

A k-d tree (k-dimensional tree) is a space-partitioning binary tree for organizing points in k-dimensional space. Invented by Jon Louis Bentley in 1975.

**How it works:** Each non-leaf node implicitly generates a splitting hyperplane that divides space into two half-spaces. The splitting dimension cycles through all k axes as you descend the tree (depth mod k). The median point is chosen at each level for a balanced tree.

**Complexity:**
- Build: O(n log n) with median-of-medians selection
- Insert: O(log n) in balanced tree
- Nearest neighbour: O(log n) average for randomly distributed points
- Range search: O(k · n^(1 - 1/k)) worst case

```js
// Construction pseudocode
function kdtree(pointList, depth) {
  const axis = depth % k; // cycle through dimensions
  const median = selectMedian(pointList, axis); // split on median
  return {
    location: median,
    left: kdtree(pointsBefore(median, axis), depth + 1),
    right: kdtree(pointsAfter(median, axis), depth + 1),
  };
}

// Nearest Neighbour Search
function nnSearch(node, target, best) {
  // 1. Recurse down the tree (go left or right based on splitting axis)
  // 2. At leaf, check distance - update best if closer
  // 3. Unwind recursion:
  //    a. Check if current node is closer than current best
  //    b. Check if hypersphere (radius = current best distance)
  //       crosses the splitting plane
  //    c. If yes → search the other subtree too
  //    d. If no  → prune that entire branch
}

// Range Search - find all points within an axis-aligned rectangle
function rangeSearch(node, range, depth) {
  if (!node) return [];
  const axis = depth % k;
  let results = [];
  if (range.contains(node.location)) results.push(node.location);
  if (range.min[axis] <= node.location[axis]) // overlap with left
    results.push(...rangeSearch(node.left, range, depth + 1));
  if (range.max[axis] >= node.location[axis]) // overlap with right
    results.push(...rangeSearch(node.right, range, depth + 1));
  return results;
}
```

**Degradation:** In high-dimensional spaces (curse of dimensionality), performance degrades toward linear search. Rule of thumb: n >> 2^k for k-d trees to be effective.

**Variants:** Implicit k-d tree, min/max k-d tree, relaxed k-d tree, K-D-B-tree, Bkd-tree. Related structures: Quadtree, Octree, Ball tree, R-tree, Vantage-point tree.

14. [Differential Privacy in Location Systems ](https://research.google/pubs/pub41892/)

Differential privacy is a mathematical framework for providing provable privacy guarantees when publishing statistical information about datasets. Google's RAPPOR (Randomized Aggregatable Privacy-Preserving Ordinal Response) applies this to location systems.

**Core concept:** A randomized algorithm M is ε-differentially private if for any two adjacent datasets D1, D2 and any output S: P(M(D1) ∈ S) ≤ e^ε · P(M(D2) ∈ S). The parameter ε (epsilon) controls the privacy-utility tradeoff — smaller ε = stronger privacy.

```js
// Differential Privacy mechanisms for location data
// 1. Laplace Mechanism - add noise proportional to sensitivity
function addLaplaceNoise(trueValue, sensitivity, epsilon) {
  const scale = sensitivity / epsilon; // b = Δf / ε
  const noise = laplaceSample(0, scale); // Lap(0, b)
  return trueValue + noise;
}

// 2. Geo-Indistinguishability - extend DP to 2D plane
// Adds planar Laplace noise to (lat, lng) coordinates
function geoIndistinguish(lat, lng, epsilon) {
  const r = gammaSample(2, 1 / epsilon); // radius
  const theta = uniformSample(0, 2 * Math.PI); // angle
  return {
    lat: lat + r * Math.cos(theta),
    lng: lng + r * Math.sin(theta),
  };
}

// 3. Exponential Mechanism - select from discrete set
// Picks a location region with probability ∝ exp(ε·utility / 2Δu)
// Useful for snapping noisy locations to valid POIs

// 4. Local Differential Privacy (LDP) - noise added on device
// Each user perturbs own data before sending to server
// Server aggregates noisy reports to get population statistics
// RAPPOR: Randomized response + Bloom filters for frequency estimation
```

**Key tradeoffs in location systems:**
- **Utility vs. Privacy:** More noise = better privacy but less accurate location services
- **Spatial granularity:** Privacy budget spent per query; frequent location updates drain budget faster
- **Composition theorem:** Total privacy loss accumulates across multiple queries (ε_total = Σ ε_i)
- **Applications:** Aggregate heatmaps, popular times, traffic estimation, census data

21. [Raft Consensus Algorithm](https://raft.github.io/)

Raft is a consensus algorithm designed to be understandable. It is equivalent to Paxos in fault-tolerance and performance but decomposed into independent subproblems: leader election, log replication, and safety.

**Core concept:** Consensus involves multiple servers agreeing on values. A cluster of 2F+1 servers tolerates F failures. Each server has a state machine and a replicated log.

```js
// Raft Node States
const states = {
  FOLLOWER: "follower",   // default state, responds to RPCs
  CANDIDATE: "candidate", // seeking votes for leadership
  LEADER: "leader",       // handles all client requests, replicates log
};

// Leader Election
// - Each node starts as FOLLOWER with a random election timeout (150-300ms)
// - If no heartbeat received → becomes CANDIDATE, increments term, votes for self
// - Sends RequestVote RPCs to all other servers
// - Wins if majority (> N/2) of votes received
// - Only one leader per term (term = logical clock)

function requestVote(candidateId, term, lastLogIndex, lastLogTerm) {
  // Grant vote if:
  // 1. Candidate's term >= voter's current term
  // 2. Voter hasn't voted for anyone else in this term
  // 3. Candidate's log is at least as up-to-date as voter's
  //    (compare lastLogTerm first, then lastLogIndex)
  return { term, voteGranted };
}

// Log Replication
// - Client sends command to leader
// - Leader appends to its log, sends AppendEntries RPCs to followers
// - Once majority acknowledges → entry is "committed"
// - Leader notifies followers of committed entries
// - Each follower applies committed entries to its state machine

function appendEntries(term, leaderId, prevLogIndex, prevLogTerm, entries, leaderCommit) {
  // Follower checks:
  // 1. term >= currentTerm (reject stale leaders)
  // 2. Log contains entry at prevLogIndex with prevLogTerm (consistency check)
  // 3. If conflict → delete conflicting entries and all after
  // 4. Append new entries, update commitIndex to min(leaderCommit, last new entry)
  return { term, success };
}

// Safety guarantees:
// - Election Safety: at most one leader per term
// - Leader Append-Only: leader never overwrites/deletes its log entries
// - Log Matching: if two logs have entry with same index+term → identical up to that point
// - Leader Completeness: committed entry appears in all future leaders' logs
// - State Machine Safety: if server applies entry at index → no other server applies different entry at same index
```

**Membership changes:** Joint consensus approach — old and new configs overlap during transition. Log compaction via snapshots (state machine state at a point, discard log entries before that).

**Production use:** etcd, CockroachDB, TiKV, Consul (hashicorp/raft), RabbitMQ (Ra), Hazelcast, ScyllaDB, many others (100+ implementations across all major languages).

22. [Paxos Algorithm Basics](https://en.wikipedia.org/wiki/Paxos_%28computer_science%29)

Paxos is a family of protocols for solving consensus in a network of unreliable processors. Proposed by Leslie Lamport, it guarantees safety (consistency) but not liveness (due to the FLP impossibility result).

**Assumptions:** Processors may fail and recover (crash-recovery model, no Byzantine faults). Messages may be lost, reordered, or duplicated but not corrupted. Requires n = 2F + 1 processors to tolerate F simultaneous failures.

```js
// Paxos Roles
const roles = {
  PROPOSER: "proposes values",           // usually the client-facing leader
  ACCEPTOR: "votes on proposals",         // maintains durable state
  LEARNER:  "learns the chosen value",    // applies decided values
};
// In practice, each server plays all three roles

// Basic Paxos - Two-Phase Protocol
// Phase 1: PREPARE / PROMISE
function phase1a_Prepare(proposer) {
  // Proposer picks unique proposal number n (must be > any previous n)
  // Sends Prepare(n) to a quorum of acceptors
  return { type: "Prepare", n };
}

function phase1b_Promise(acceptor, prepareMsg) {
  // If n > any previously seen proposal number:
  //   Promise to ignore all proposals < n
  //   Reply with highest previously accepted (n_a, v_a) if any
  // Else: ignore or send Nack
  return { type: "Promise", n: prepareMsg.n, accepted: { n_a, v_a } };
}

// Phase 2: ACCEPT / ACCEPTED
function phase2a_Accept(proposer, promises) {
  // If received Promise from quorum:
  //   If any acceptor reported a previously accepted value:
  //     Use the value with the highest proposal number (v = v_z)
  //   Else: proposer is free to choose any value (v = v_x)
  //   Send Accept!(n, v) to quorum
  return { type: "Accept", n, v };
}

function phase2b_Accepted(acceptor, acceptMsg) {
  // If acceptor has NOT promised to ignore this n:
  //   Accept the value, send Accepted(n, v) to proposer + all learners
  // Else: ignore
  // Consensus achieved when majority of acceptors accept same n
  return { type: "Accepted", n, v };
}

// Multi-Paxos Optimization
// If leader is stable, skip Phase 1 for subsequent instances
// Reduces message delay from 4 to 2 (just Accept→Accepted)
// Assigns incrementing instance number I to each command

// Paxos Variants:
// - Multi-Paxos:       skip Phase 1 with stable leader (2 msg delays)
// - Cheap Paxos:       F+1 main + F auxiliary processors
// - Fast Paxos:        2 msg delays (client→acceptors direct), needs 3F+1
// - Generalized Paxos: exploits commutativity of operations
// - Byzantine Paxos:   adds Verify phase for Byzantine fault tolerance
```

**Safety properties:** Validity (only proposed values chosen), Agreement (no two learners learn different values). Liveness not guaranteed (FLP impossibility — can't have safety + liveness + fault tolerance simultaneously).

**Production use:** Google Chubby/Spanner/Megastore, Microsoft Autopilot, Apache Cassandra (lightweight transactions), Amazon DynamoDB, Neo4j, MariaDB Xpand, Apache ZooKeeper (ZAB variant).

23. [Google S2 Geometry Library](https://s2geometry.io/)

S2 is a library for spherical geometry. Unlike traditional GIS (flat 2D projections), S2 represents all data on a 3D sphere — no seams, singularities, or discontinuities. Name derived from the mathematical notation S² (unit sphere).

**S2 Cell Hierarchy:** The sphere is decomposed by projecting 6 cube faces onto the unit sphere, then recursively subdividing each face into 4 children (quadtree). 31 levels (0-30), with level 30 "leaf cells" ≈ 1cm on Earth's surface. Total leaf cells: 6 × 4^30.

```js
// S2CellId: 64-bit identifier encoding face + position on Hilbert curve
// Format: [face:3 bits][child_0..child_k: 2 bits each][1][zeros]
// The trailing 1-bit marks the level; remaining bits are zero
// Example: 010 10 1 000...0  = subcell 2 of face cell 2 (level 1)

// S2 Coordinate Systems (in order of transformation)
// 1. (cellid)       → 64-bit Hilbert curve position (spatial indexing key)
// 2. (face, i, j)   → leaf cell coordinates (2^30 × 2^30 grid per face)
// 3. (face, s, t)   → [0,1]×[0,1] real coordinates on face
// 4. (face, u, v)   → [-1,1]×[-1,1] (nonlinear transform for uniform cell size)
// 5. (x, y, z)      → unit vector on sphere (the actual S2Point)

// Key advantage: cells numbered along Hilbert space-filling curve
// → nearby S2CellIds = nearby cells on sphere (great for spatial indexes)

// S2CellId operations (C++ API, similar in Go/Java/Python)
S2CellId.FromFacePosLevel(face, pos, level); // construct from components
cellId.face();           // which cube face (0..5)
cellId.level();          // subdivision level (0..30)
cellId.parent();         // parent cell
cellId.child(position);  // child at position 0..3
cellId.contains(other);  // containment test
cellId.range_min();      // smallest leaf cell within this cell
cellId.range_max();      // largest leaf cell within this cell
cellId.next();           // next cell along Hilbert curve
cellId.prev();           // previous cell along Hilbert curve
cellId.GetEdgeNeighbors(neighbors); // 4 adjacent cells

// S2RegionCoverer: approximate any region as a union of S2Cells
const options = { maxCells: 8, minLevel: 0, maxLevel: 30 };
const covering = S2RegionCoverer.GetCovering(region, options);
// Returns S2CellUnion — sorted, non-overlapping set of cell ids
// Used for spatial indexing: store covering cell ids in database index

// S2CellUnion operations
union.Contains(cellId);     // does the union contain this cell?
union.Intersects(cellId);   // does the union intersect this cell?
union1.Union(union2);       // set union
union1.Intersection(union2);// set intersection
```

28. [Cloudflare Geo Load Balancing](https://developers.cloudflare.com/load-balancing/)

Cloudflare Load Balancing distributes traffic across multiple origins based on geographic proximity, health checks, and steering policies — ensuring low latency and high availability.

```js
// Cloudflare Geo Load Balancing Architecture

// Core concepts:
// Pool   = group of origin servers (e.g., us-east pool, eu-west pool)
// Monitor = health check configuration (HTTP, HTTPS, TCP, ICMP)
// Rule   = traffic steering logic (geo, random, hash, proximity, etc.)

// 1. Geo Steering — route by client region
const geoSteeringPolicy = {
  steering: "geo",
  pools: {
    "NA":  ["pool-us-east", "pool-us-west"],  // North America → US pools
    "EU":  ["pool-eu-west", "pool-eu-central"],
    "AS":  ["pool-ap-southeast", "pool-ap-northeast"],
    "OC":  ["pool-ap-southeast"],              // Oceania → closest Asian pool
    "default": ["pool-us-east"],               // fallback for unmapped regions
  },
  // Cloudflare maps client IP → region → route to nearest pool
  // Failover: if primary pool unhealthy → next pool in list
};

// 2. Proximity Steering (dynamic) — route to nearest healthy origin
// Uses Cloudflare edge PoP location + origin coordinates
// Automatically picks lowest-latency origin based on real-time RTT
const proximityPolicy = {
  steering: "proximity",
  origins: [
    { name: "us-east", lat: 39.0, lng: -77.5 },
    { name: "eu-west", lat: 51.5, lng: -0.1 },
    { name: "ap-se",   lat: 1.3,  lng: 103.8 },
  ],
};

// 3. Health Checks — continuous origin monitoring
const healthCheck = {
  type: "HTTPS",
  path: "/health",
  interval: 60,          // check every 60 seconds
  timeout: 5,            // 5 second timeout
  retries: 2,            // retry twice before marking unhealthy
  expected_codes: "2xx", // 200-299 = healthy
  follow_redirects: true,
  // Checked from multiple Cloudflare PoPs worldwide
};

// 4. Session Affinity — sticky sessions by geo
const sessionAffinity = {
  type: "cookie",          // or "ip_cookie", "header"
  ttl: 1800,               // 30 minutes
  // Ensures user stays on same origin within a session
  // Even with geo steering, prevents mid-session origin switches
};

// 5. Load Balancing Rules (custom traffic steering)
// Match by URI, header, cookie, source IP, geo, etc.
// Example: route /api/v2/* to canary pool, everything else to prod
const rule = {
  condition: 'http.request.uri.path matches "^/api/v2/"',
  action: { pool: "canary-pool", weight: 10 },
  // Weighted pools for gradual rollouts (canary deployments)
};

// Traffic flow:
// Client → Cloudflare Edge (anycast to nearest PoP)
//   → Geo/Proximity steering selects pool
//   → Health check filters unhealthy origins
//   → Load balance within pool (round-robin or weighted)
//   → Request forwarded to selected origin
```

**Key features:** 300+ global PoPs, sub-second failover, SSL/TLS termination at edge, DDoS protection included, Argo Smart Routing (optimized paths), integrated with Workers (edge compute) for geo-aware logic.
