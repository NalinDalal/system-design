# Web Crawler

A web crawler is known as a robot or spider.
It is widely used by search engines to discover new or updated content on the web - pages, images, videos, PDFs, and more.
Crawlers are used for search indexing, archiving, monitoring, and data collection.

## Purposes

- **Search engine indexing**: A crawler collects web pages to create a local index for search engines.
  For example, Googlebot is the web crawler behind the Google search engine.

- **Web archiving**: This is the process of collecting information from the web to preserve data for future uses. 
  For instance, many national libraries run crawlers to archive web sites. Notable examples are the US Library of Congress and the EU web archive.

- **Web mining**: The explosive growth of the web presents an unprecedented opportunity for data mining. 
  Web mining helps to discover useful knowledge from the internet. 
  For example, top financial firms use crawlers to download shareholder meetings and annual reports to learn key company initiatives.

- **Web monitoring**: The crawlers help to monitor copyright and trademark infringements over the Internet. 
  For example, Digimarc [3] utilizes crawlers to discover pirated works and reports.


## Design scope

Basic crawler loop:

1. Given a set of seed URLs, download the pages.
2. Extract URLs from downloaded pages.
3. Add unseen URLs to the frontier and repeat.

Requirements example:

- Target: search engine indexing
- Scale: ~1 billion pages/month
- Content: HTML (for this design)
- Retention: store content for up to 5 years
- Scalability: support parallel downloaders
- Robustness: handle timeouts, server errors, crashes
- Extensibility: support new content types with minimal changes

## Back-of-Envelope estimation

- 1 billion pages / month
- QPS = 1,000,000,000 / 30 / 24 / 3600 ≈ 400 pages/sec
- Peak QPS ≈ 2 × QPS ≈ 800 pages/sec
- Avg page size ≈ 500 KB → monthly storage ≈ 500 TB
- For 5 years: 500 TB × 12 × 5 ≈ 30 PB

### Seed URLs

Seed URLs bootstrap the crawl and should provide broad coverage (by topic, domain, locality, etc.).

### URL Frontier

Frontier states: `to-be-downloaded` and `downloaded`. The frontier enforces politeness, prioritization, and freshness.

### HTML downloader

Responsible for fetching pages over HTTP(S). Works with a DNS resolver and respects `robots.txt`.

### DNS resolver

Resolves hostnames to IPs. Use a cache to reduce latency.

### Content parser

Validates and normalizes content to avoid storing malformed pages.

### Content seen

Detect duplicate content using hashes/fingerprints (e.g., Rabin fingerprints, Bloom filters). Hash comparison is efficient at scale.

### Content storage

Store HTML on disk; keep popular content cached in memory for low-latency access.

### URL extractor

Parses links from HTML and normalizes relative URLs to absolute (e.g., add https://en.wikipedia.org as needed).

```mermaid

```

### URL Filter

Exclude undesired content types, known bad extensions, error pages, and blacklisted sites.

### URL seen?

Maintain a disk-backed + in-memory index to prevent re-adding already-seen URLs.

### URL storage

Persist visited URLs for analytics and to support restarts.

## Deep dive

- DFS vs BFS: the web is a directed graph. BFS (queue-based) is commonly used but naive BFS can over-request single hosts and ignores URL priority. Prioritization should bias the frontier.
- URL frontier: design for politeness, prioritization, and freshness.
- HTML downloader: parallel, host-aware workers with per-host queues and delays.
- Robustness: checkpoint state, retry/backoff, and exception handling.
- Extensibility: support plug-in modules for new content types.

### Politeness

Avoid overloading a single host by routing URLs into per-host queues. Each worker consumes from a queue dedicated to a single host; enforce delays between requests to the same host.

```mermaid
flowchart TD
  subgraph Router["Queue router"]
  end

  Router --> M[(Mapping Table)]
  Router --> Q1["Queue b1"]
  Router --> Q2["Queue b2"]
  Router --> Qn["Queue bn"]

  Q1 --> Selector["Queue selector"]
  Q2 --> Selector
  Qn --> Selector

  Selector --> W1["Worker thread 1"]
  Selector --> W2["Worker thread 2"]
  Selector --> Wn["Worker thread N"]

  classDef db fill:#1f77b4,color:#fff,stroke:#333,stroke-width:1px;
  classDef worker fill:#2ca02c,color:#fff,stroke:#333,stroke-width:1px;
  classDef queue fill:#fff,stroke:#333,stroke-width:1px;

  class M db;
  class W1,W2,Wn worker;
  class Q1,Q2,Qn queue;
```

- Queue router: It ensures that each queue (b1, b2, … bn) only contains URLs from the same host.
- Mapping table: It maps each host to a queue.
- FIFO queues b1, b2 to bn: Each queue contains URLs from the same host.
- Queue selector: Each worker thread is mapped to a FIFO queue, and it only downloads URLs from that queue. The queue selection logic is done by the Queue selector.
- Worker thread 1 to N. A worker thread downloads web pages one by one from the same host. A delay can be added between two download tasks.

### Prioritization

Use a `Prioritizer` to score URLs (PageRank, traffic, update frequency, domain importance) and assign them to priority queues. The front queue selector biases selection toward higher-priority queues.

say search for apple, how does it keeps apple.com on priority?
We prioritize URLs based on usefulness, which can be measured by PageRank [10], website traffic, update frequency, etc.
`Prioritizer` is the component that handles URL prioritization.

```mermaid

    %% Queues

    P --> F2["Queue f2"]
    P --> Fn["Queue fn"]

    %% Queue selector
    F1 --> QS[Queue selector]
    F2 --> QS
    Fn --> QS

    QS --> Output["output URLs"]

    %% Styling
    classDef queue fill:#fff,stroke:#333,stroke-width:1px;
    classDef process fill:#bbf,stroke:#333,stroke-width:1px;
    classDef io fill:#eee,stroke:#333,stroke-width:1px;

    class F1,F2,Fn queue;
    class P,QS process;

```

- Prioritizer: It takes URLs as input and computes the priorities.
- Queue f1 to fn: Each queue has an assigned priority. Queues with high priority are selected with higher probability.
- Queue selector: Randomly choose a queue with a bias towards queues with higher priority.

### Freshness

Recrawl based on observed update frequency; prioritize important pages for more frequent recrawls.

### Storage for the URL frontier

Frontier size may be hundreds of millions of URLs. Use a hybrid approach: disk-backed storage with in-memory buffers for enqueue/dequeue operations to avoid disk bottlenecks.

## HTML Downloader
It downloads web pages from internet via HTTP protocol.

## Robots.txt

Respect `robots.txt` (Robots Exclusion Protocol- standard to communicate with crawlers) and cache parsed rules to avoid repeated fetches. Example: [Amazon robots.txt](https://www.amazon.com/robots.txt)

## Performance optimizations

1. Distributed crawling: partition URL space across downloaders.
2. DNS cache: avoid repeated DNS lookups (can be ~200 ms).
3. Locality: prefer geographically close servers when possible.
4. Short timeouts and aggressive retries to avoid long blocking.

## Robustness

- Consistent hashing: distribute URL buckets among downloaders and reduce rebalancing.
- Persistent checkpoints: save crawl state to allow restart after crashes.
- Strong exception handling and data validation.

## Extensibility

Design extension points (modules) for format-specific downloaders/parsers (PNG downloader, link extractor, monitors, etc.).

## Detect and avoid problematic content

1. Redundant content: ~30% duplicates — use content fingerprints and bloom filters.
2. Spider traps: limit URL depth/length and use heuristics and manual blacklists.
3. Data noise: filter out ads and low-value content with anti-spam heuristics.

## Extra notes

- Server-side (dynamic) rendering can be required to extract JavaScript-generated links; use headless rendering selectively.
- Use anti-spam filters to avoid low-quality pages.
- Database replication and sharding improve availability and scalability.
- Keep crawler servers stateless to simplify horizontal scaling.
- Instrumentation and analytics are essential for tuning crawl policies.
