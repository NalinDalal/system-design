## Step 1 - Understand the Problem and Establish Design Scope

The hotel reservation system is complicated and its components vary based on business use cases. Before diving into the design, we should ask the interviewer clarification questions to narrow down the scope.

**Candidate:** What is the scale of the system?
**Interviewer:** Let's assume we are building a website for a hotel chain that has 5,000 hotels and 1 million rooms in total.

**Candidate:** Do customers pay when they make reservations or when they arrive at the hotel?
**Interviewer:** For simplicity, they pay in full when they make reservations.

**Candidate:** Do customers book hotel rooms through the hotel's website only? Do we need to support other reservation options such as phone calls?
**Interviewer:** Let's assume people could book a hotel room through the hotel website or app.

**Candidate:** Can customers cancel their reservations?
**Interviewer:** Yes.

**Candidate:** Are there any other things we need to consider?
**Interviewer:** Yes, we allow 10% overbooking. In case you do not know, overbooking means the hotel will sell more rooms than they actually have. Hotels do this in anticipation that some customers will cancel their reservations.

**Candidate:** Since we have limited time, I assume the hotel room search is not in scope. We focus on the following features:

- Show the hotel-related page.
- Show the hotel room-related detail page.
- Reserve a room.
- Admin panel to add/remove/update hotel or room info.
- Support the overbooking feature.

**Interviewer:** Sounds good.

**Interviewer:** One more thing, hotel prices change dynamically. The price of a hotel room depends on how full the hotel is expected to be on a given day. For this interview, we can assume the price could be different each day.

**Candidate:** I'll keep this in mind.

### Non-functional Requirements

- **Support high concurrency.** During peak season or big events, some popular hotels may have a lot of customers trying to book the same room.
- **Moderate latency.** It's ideal to have a fast response time when a user makes the reservation, but it's acceptable if the system takes a few seconds to process a reservation request.

### Back-of-the-envelope Estimation

- 5,000 hotels and 1 million rooms in total.
- Assume 70% of the rooms are occupied and the average stay duration is 3 days.
- Estimated daily reservations: `1,000,000 × 0.7 / 3 ≈ 233,333` (rounding up to ~240,000)
- Reservations per second: `240,000 / 86,400 ≈ 3 TPS`

As we can see, the average reservation transaction per second (TPS) is not high.

#### QPS Estimation (Funnel Model)

There are three steps in a typical customer flow:

1. **View hotel/room detail page** — Users browse this page (query).
2. **View the booking page** — Users can confirm booking details such as dates, number of guests, payment information before booking (query).
3. **Reserve a room** — Users click on the "book" button to book the room and the room is reserved (transaction).

Assume ~10% of users reach the next step (90% drop off at each stage). No prefetching is implemented. Working backward from the final reservation TPS of 3:

```
View hotel/room detail page  →  QPS ≈ 300
Order booking page            →  QPS ≈ 30
Reserve rooms                 →  QPS ≈ 3
```

### API Design

We explore the API design for the hotel reservation system using RESTful conventions.

> **Note:** This chapter focuses on the design of a hotel reservation system. For a complete hotel website, the design needs to provide intuitive features for customers to search for rooms based on a large array of criteria. The APIs for these search features, while important, are not technically challenging. They are out of scope for this chapter.

> `reservationId` is generated by a globally unique ID generator (idempotency key) to avoid double booking.

### Data Model

Before we decide which database to use, let's take a close look at the data access patterns. For the hotel reservation system, we need to support the following queries:

- **Query 1:** View detailed information about a hotel.
- **Query 2:** Find available types of rooms given a date range.
- **Query 3:** Record a reservation.
- **Query 4:** Look up a reservation or past history of reservations.

From the back-of-the-envelope estimation, we know the scale of the system is not large but we need to prepare for traffic surges during big events. With these requirements in mind, we choose a **relational database** because:

- A relational database works well with **read-heavy and write less frequently** workflow. This is because the number of users who visit the hotel website/apps is a few orders of magnitude higher than those who actually make reservations. NoSQL databases are generally optimized for writes and the relational database works well enough for read-heavy workflow.
- A relational database provides **ACID** (atomicity, consistency, isolation, durability) guarantees. ACID properties are important for a reservation system. Without those properties, it's not easy to prevent problems such as negative balance, double charge, double reservations, etc. ACID properties make application code a lot simpler and make the whole system easier to reason about. A relational database usually provides these guarantees.
- A relational database can easily **model the data**. The structure of the business data is very clear and the relationship between different entities (`hotel`, `room`, `room_type`, etc) is stable. This kind of data model is easily modeled by a relational database.

Most attributes are self-explanatory. We will only explain the `status` field in the reservation table. The `status` field can be in one of these states: **pending**, **paid**, **refunded**, **canceled**, **rejected**. The state machine is:

```
                  Pending
                /    |    \
          Canceled  Paid  Rejected
                     |
                  Refunded
```

> This schema design has a major issue. This data model works for companies like Airbnb as `room_id` (might be called `listing_id`) is given when users make reservations. However, this isn't the case for hotels. A user actually reserves a **type of room** in a given hotel instead of a specific room. For instance, a room type can be a standard room, king-size room, queen-size room with two queen beds, etc. Room numbers are given when the guest checks in and not at the time of the reservation. We need to update our data model to reflect this new requirement. See "Improved data model" in the deep dive section for more details.

### High-level Design

We use the **microservice architecture** for this hotel reservation system. Over the past few years, microservice architecture has gained great popularity. Companies that use microservice include Amazon, Netflix, Uber, Airbnb, Twitter, etc.

Our design is modeled with the microservice architecture and the high-level design diagram is as follows:

```
                        Public    Private
  User ──────┐          │           │
             CDN        │         Admin
  External   │          │           │
  ─────────────────────────────────────────
  Internal   │          │           │
             └──> Public API    Internal API
                  Gateway          │
                    │              │
        ┌───────┬───┴────┬─────────┼──────────┐
        │       │        │         │          │
     Hotel   Rate   Reservation  Payment   Hotel Mgmt
     Service Service  Service    Service    Service
        │       │        │         │
      Cache   Hotel   Rate DB  Reservation  Payment
              DB               DB           DB
```

We will briefly go over each component of the system from top to bottom:

- **User:** a user books a hotel room on their mobile phone or computer.
- **Admin (hotel staff):** authorized hotel staff perform administrative operations such as refunding a customer, canceling a reservation, updating room information, etc.
- **CDN (content delivery network):** for better load time, CDN is used to cache all static assets, including javascript bundles, images, videos, HTML, etc.
- **Public API Gateway:** this is a fully managed service that supports rate limiting, authentication, etc. The API gateway is configured to direct requests to specific services based on the endpoints. For example, requests to load the hotel homepage are directed to the hotel service and requests to book a hotel room are routed to the reservation service.
- **Internal APIs:** those APIs are only available for authorized hotel staff. They are accessible through internal software or websites. They are usually further protected by a VPN (virtual private network).
- **Hotel Service:** this provides detailed information on hotels and rooms. Hotel and room data are generally static, so can be easily cached.
- **Rate Service:** this provides room rates for different future dates. An interesting fact about the hotel industry is that the price of a room depends on how full the hotel is expected to be for a given day.
- **Reservation Service:** receives reservation requests and reserves the hotel rooms. This service also tracks room inventory as rooms are reserved or reservations are canceled.
- **Payment Service:** executes payment from a customer and updates the reservation status to paid once a payment transaction succeeds, or rejected if the transaction fails.
- **Hotel Management Service:** only available to authorized hotel staff. Hotel staff are eligible to use the following features: view the record of an upcoming reservation, reserve a room for a customer, cancel a reservation, etc.

#### Connections Between Services

For clarity, the high-level diagram omits many arrows of interactions between microservices. For example, there should be an arrow between Reservation service and Rate service. Reservation service queries Rate service for room rates. This is used to compute the total room charge for a reservation. Another example is that there should be many arrows connecting the Hotel Management Service with most of the other services. When an admin makes changes via Hotel Management Service, the requests are forwarded to the actual service owning the data, to handle the changes.

```
  Reservation ──────────┐     Hotel Management
    Service              │       Service
       │                 │      / | \  \
    Rate Service    Hotel Service  Rate  Reservation  ...
                                Service   Service
```

For production systems, inter-service communication often employs a modern and high-performance remote procedure call (RPC) framework like **gRPC**. There are many benefits to using such frameworks.

### Improved Data Model

As mentioned earlier, the initial schema has a major issue: guests reserve a **type of room** rather than a specific room. A user books a "standard room" or a "king-size room" — the actual room number is assigned at check-in.

To reflect this, we redesign the schema around `room_type_inventory` instead of individual rooms.

#### Updated Schema

**hotel**

| Column    | Type    |
| --------- | ------- |
| hotel_id  | PK      |
| name      | varchar |
| address   | varchar |
| location  | varchar |

**room_type**

| Column          | Type    |
| --------------- | ------- |
| room_type_id    | PK      |
| hotel_id        | FK      |
| name            | varchar |
| description     | varchar |
| max_occupancy   | int     |

**room_type_inventory** (tracks availability per room type per day)

| Column              | Type    |
| ------------------- | ------- |
| hotel_id            | PK      |
| room_type_id        | PK      |
| date                | PK (date) |
| total_inventory     | int     |
| total_reserved      | int     |

**room_type_rate**

| Column        | Type    |
| ------------- | ------- |
| hotel_id      | PK      |
| room_type_id  | PK      |
| date          | PK      |
| rate          | decimal |

**reservation**

| Column          | Type     |
| --------------- | -------- |
| reservation_id  | PK (uuid) |
| hotel_id        | FK       |
| room_type_id    | FK       |
| start_date      | date     |
| end_date        | date     |
| status          | enum     |
| guest_id        | FK       |

**guest**

| Column     | Type    |
| ---------- | ------- |
| guest_id   | PK      |
| first_name | varchar |
| last_name  | varchar |
| email      | varchar |

The key table is `room_type_inventory`. It stores inventory data for every `(hotel_id, room_type_id, date)` combination. Here:

- `total_inventory` — the total number of rooms of this type in the hotel (set by the admin).
- `total_reserved` — the total number of rooms reserved for that day.

A room is available if: **`total_inventory - total_reserved > 0`**

To support **10% overbooking**, the check becomes:

```
total_inventory × 1.10 - total_reserved > 0
```

#### Reservation Flow with the New Model

When a user makes a reservation for a given `(hotel_id, room_type_id, [start_date, end_date])`:

1. Check `room_type_inventory` for each date in `[start_date, end_date)` to ensure availability.
2. If all dates have availability, atomically increment `total_reserved` for each affected date.
3. Insert a new row in the `reservation` table with status = `pending`.
4. Proceed to payment.

#### Storing Reservation Data at Scale

Most queries on the `reservation` table are for active/upcoming reservations. Historical data can be archived to a cold storage or separate data warehouse for analytics, keeping the hot table small and performant.

### Concurrency Issues

The most critical concurrency issue in a reservation system is the **double booking** problem — two users trying to book the last available room at the same time.

#### Problem Illustration

Consider a scenario where only 1 room remains (`total_inventory - total_reserved = 1`):

```
User A: reads total_reserved = 99            (available = 1)
User B: reads total_reserved = 99            (available = 1)
User A: updates total_reserved = 100         (success)
User B: updates total_reserved = 100         (ALSO success — DOUBLE BOOKING!)
```

Both users see availability and both succeed. This is a classic **lost update** / **race condition**.

#### Solution 1: Pessimistic Locking

Use a `SELECT ... FOR UPDATE` to lock the rows during the transaction:

```sql
BEGIN;

SELECT total_reserved
FROM room_type_inventory
WHERE hotel_id = ? AND room_type_id = ? AND date = ?
FOR UPDATE;

-- Application checks: if total_reserved < total_inventory * 1.10
UPDATE room_type_inventory
SET total_reserved = total_reserved + 1
WHERE hotel_id = ? AND room_type_id = ? AND date = ?;

COMMIT;
```

**Pros:**
- Prevents race conditions.
- Easy to implement and reason about.

**Cons:**
- Deadlocks can occur if multiple rows are locked in different orders.
- Reduces throughput because requests are serialized on the locked rows.
- Not suitable for highly concurrent systems.

#### Solution 2: Optimistic Locking (Versioning)

Add a `version` column to `room_type_inventory`. Each update checks the version:

```sql
-- Read
SELECT total_reserved, version
FROM room_type_inventory
WHERE hotel_id = ? AND room_type_id = ? AND date = ?;

-- Update (only succeeds if version hasn't changed)
UPDATE room_type_inventory
SET total_reserved = total_reserved + 1, version = version + 1
WHERE hotel_id = ? AND room_type_id = ? AND date = ?
  AND version = ?;
```

If the `UPDATE` returns 0 rows affected, the user retries.

**Pros:**
- No locks held, higher throughput.
- Works well when contention is low.

**Cons:**
- Under high contention, many retries degrade performance.
- Users may experience frustrating failures during peak times.

#### Solution 3: Database Constraints (Recommended)

Use a simple atomic SQL statement with a `WHERE` clause check:

```sql
UPDATE room_type_inventory
SET total_reserved = total_reserved + 1
WHERE hotel_id = ? AND room_type_id = ? AND date = ?
  AND total_reserved + 1 <= FLOOR(total_inventory * 1.10);
```

If 0 rows are updated, the room is not available. This approach:

**Pros:**
- Simple, no extra version column needed.
- The database engine handles atomicity — no race condition.
- Works well under both low and high contention.
- Minimal application-side complexity.

**Cons:**
- Relies on the database handling the constraint correctly (all major RDBMS do).

This is the **recommended approach** for its simplicity and correctness. The `total_reserved` update is atomic at the database level, so two concurrent transactions cannot both succeed if only one slot remains.

### Scaling the System

Even though the TPS (~3) is low, we should design for peak load and future growth.

#### Database Sharding

The `room_type_inventory` and `reservation` tables can be sharded by `hotel_id`. This gives us:

- **Even distribution:** different hotels' data lives on different shards.
- **Query locality:** all queries for a single hotel hit the same shard.

With 5,000 hotels, even 16 shards gives ~312 hotels per shard, which is more than enough.

#### Caching

- **Hotel & room data** are mostly static — cache aggressively in Redis/Memcached with a long TTL.
- **Rate data** changes daily — cache with shorter TTL or invalidate on update.
- **Inventory data** is highly dynamic (changes on every reservation) — caching is risky because stale data can lead to overbooking or false "sold out" signals. It's better to query the database for inventory checks. However, we can cache a "sold out" flag: once a room type is fully booked for a date, cache that fact. This avoids unnecessary DB queries for clearly unavailable dates.

#### Read Replicas

Set up read replicas for the database to handle the high read QPS (hotel detail pages at ~300 QPS). Writes (reservations at ~3 TPS) go to the primary. Read replicas serve:
- Hotel detail pages
- Room type listings
- Rate lookups
- Reservation history queries

#### CDN and Static Content

All hotel images, descriptions, and static page content should be served via CDN (as mentioned in the high-level design).

#### Service-Level Scaling

Each microservice can be scaled independently:
- **Hotel Service:** scale horizontally behind a load balancer; add more cache nodes as needed.
- **Rate Service:** moderate load, can scale with read replicas.
- **Reservation Service:** the most critical; scale carefully with database sharding + connection pooling.
- **Payment Service:** usually delegates to a third-party payment processor, so the bottleneck is the external API.

### Resolving Data Inconsistency in the Microservice Architecture

In a monolithic architecture, a single database transaction can span multiple tables atomically. In a microservice architecture, each service owns its own database. This creates challenges when a business operation spans multiple services.

For example, when a user makes a reservation:
1. **Reservation Service** creates a reservation and updates inventory.
2. **Payment Service** charges the customer.
3. **Reservation Service** updates status to "paid."

If step 2 fails after step 1 succeeds, we have an **inconsistent state** — inventory is decremented but no payment was made.

#### Solution 1: Two-Phase Commit (2PC)

A distributed transaction protocol where a coordinator ensures all participants either commit or abort.

**Pros:** Strong consistency.
**Cons:** Slow, blocking, not fault-tolerant. A single participant failure can block the entire transaction. **Not recommended** for microservices at scale.

#### Solution 2: Saga Pattern (Recommended)

A saga is a sequence of local transactions where each step has a corresponding **compensating transaction** (rollback action) if a subsequent step fails.

**Reservation Saga:**

| Step | Action                          | Compensating Action            |
| ---- | ------------------------------- | ------------------------------ |
| 1    | Reserve room (update inventory) | Release room (decrement reserved) |
| 2    | Process payment                 | Refund payment                 |
| 3    | Update reservation to "paid"    | Update reservation to "canceled" |
| 4    | Send confirmation notification  | Send cancellation notification |

If step 2 (payment) fails, step 1's compensating action runs: the inventory is restored.

**Two Saga Coordination Approaches:**

1. **Choreography:** Each service emits events and other services react. Simple but harder to debug as the number of services grows.

```
Reservation Service → "RoomReserved" event
Payment Service (listens) → processes payment → "PaymentCompleted" event
Reservation Service (listens) → updates status to "paid"
```

2. **Orchestration (recommended):** A central **Saga Orchestrator** coordinates the steps. It tells each service what to do and handles compensations on failure.

```
Saga Orchestrator
    │
    ├──> Reservation Service: reserve room
    ├──> Payment Service: process payment
    ├──> Reservation Service: mark as paid
    └──> Notification Service: send confirmation
```

**Pros of Orchestration:**
- Clear, centralized control flow — easy to understand and debug.
- Each service doesn't need to know about other services.
- Easy to add new steps.

**Cons:**
- The orchestrator is a single point of complexity (not failure, since it can be made highly available).

#### Idempotency

To handle retries safely (e.g., network timeouts, duplicate messages), every API must be **idempotent**. The `reservationId` generated client-side acts as an **idempotency key**:

- If the server receives a duplicate `reservationId`, it returns the existing reservation instead of creating a new one.
- This prevents double booking and double charging.

```sql
-- Use UPSERT or check-before-insert
INSERT INTO reservation (reservation_id, hotel_id, room_type_id, ...)
VALUES (?, ?, ?, ...)
ON CONFLICT (reservation_id) DO NOTHING;
```
